psychotherapy,

2.  Core  Platform  Definition  A  multilingual,  LMS-integrated  AI  clinical-skills  simulation  platform  for  psychotherapy  and  LI-CBT/ACT  training,  with  hierarchical  access  control,  guided  case  authoring,  ground-truth  patient  boundaries,  dynamic  emotional-state  simulation,  competency-linked  rubrics,  graph-based  dashboards,  transcript-based  evaluation,  student  debriefing,  technical  support  escalation,  academic-safety  protection,  usage  governance,  and  longitudinal  development  tracking  across  student,  course,  programme,  and  college  levels.    
3.  Core  Educational  and  Organisational  Hierarchy
---- page 3 ----
The  platform  must  recognise  the  following  hierarchy:  
College    ↓  Programme    ↓  Course    ↓  Simulation  Assignment    ↓  Student  Attempt    ↓  Conversation  +  Evaluation  +  Rubric  Scores   
This  hierarchy  controls:  
●  access  permissions;  ●  dashboards;  ●  reporting;  ●  rubric  aggregation;  ●  competency  analytics;  ●  credit  allocation;  ●  LMS  integration;  ●  technical  support  routing;  ●  grade  return;  ●  retry  permissions;  ●  longitudinal  student  tracking.

5.  Access  Control  Model  
The  system  should  use  both:  
1.  Role-based  access  control  —  what  type  of  user  the  person  is.  2.  Scope-based  access  control  —  where  in  the  hierarchy  the  user  has  access.

6.  Integration  Model  
The  simulator  should  function  as  an  independent  platform  with  multiple  entry  points.  
Entry  Points  ├──  Canvas  LMS  —  self-hosted  ├──  Moodle
---- page 7 ----
└──  Secure  standalone  link   
The  same  simulation  engine,  dashboards,  rubrics,  evaluations,  multilingual  logic,  support  
system,
 
and
 
credit
 
model
 
should
 
work
 
regardless
 
of
 
entry
 
method.
 
 
6.1  Canvas  LMS  Integration  
Canvas  integration  is  a  core  requirement.  
Preferred  approach:  
Canvas  self-hosted  LMS    ↓  LTI  1.3  launch    ↓  Simulator  platform    ↓  Student  completes  simulation    ↓  Evaluation  generated    ↓  Optional  grade  /  feedback  return  to  Canvas   
Canvas  integration  should  support:  
●  LTI  1.3  launch;  ●  Canvas  user  identification;  ●  Canvas  course  context;  ●  Canvas  role  detection;  ●  Canvas  assignment  context;  ●  Deep  Linking  for  selecting  simulations;  ●  Assignment  and  Grade  Services  for  grade  return;  ●  optional  Names  and  Role  Provisioning  for  roster  sync;  ●  diagnostic  logging  for  failed  launches  and  grade-sync  problems;  ●  technical  support  escalation  linked  to  Canvas  course  and  assignment  context;  ●  technical-failure  status  visible  to  teachers.  
 
6.2  Moodle  Integration  
Moodle  should  be  supported  through  LTI.  
Moodle  course  activity
---- page 8 ----
↓  LTI  launch    ↓  Simulator  platform    ↓  Simulation  attempt    ↓  Evaluation    ↓  Optional  Moodle  grade  return   
Moodle  integration  should  support:  
●  LTI  launch;  ●  Moodle  user  identification;  ●  Moodle  course  context;  ●  Moodle  role  detection;  ●  simulation  assignment  context;  ●  grade  return  where  configured;  ●  diagnostic  logging  for  launch  and  grade-sync  failures;  ●  technical  support  escalation  linked  to  Moodle  course/activity  context;  ●  technical-failure  status  visible  to  teachers.  
 
6.3  Secure  Link  Mode  
The  platform  should  also  support  secure  standalone  link  access.  
This  is  useful  for:  
●  pilots;  ●  workshops;  ●  external  training;  ●  demos;  ●  colleges  without  LMS  integration;  ●  private  cohorts;  ●  future  student-purchased  practice.  
Secure  link  access  can  use:  
●  invitation  links;  ●  access  codes;  ●  email  login;  ●  magic  link  login;  ●  CSV  roster  upload;  ●  manual  enrolment.
---- page 9 ----
Even  in  secure  link  mode,  the  same  hierarchy  should  remain:  
College    ↓  Programme    ↓  Course  /  Group    ↓  Simulation    ↓  Student  Attempt    
7.  LMS  Launch  Resilience  
The  Integration  Layer  must  be  designed  for  failure-tolerant  LMS  launch  handling.  
Some  LTI  errors  may  occur  before  the  main  React  application  loads.  
Examples:  
Possible  LTI  launch  failures:  ├──  invalid  LTI  signature  ├──  expired  token  ├──  missing  deployment/client  key  ├──  incorrect  Canvas/Moodle  configuration  ├──  bad  redirect  URI  ├──  missing  course/assignment  context  ├──  role/context  claim  mismatch  └──  course  copy  /  assignment  mapping  problem    
7.1  Fallback  LTI  Error  Page  
Because  some  LTI  errors  occur  before  the  normal  frontend  application  can  boot,  the  
backend
 
must
 
provide
 
a
 
lightweight
 
fallback
 
error
 
page.
 
LTI  launch  attempt    ↓  Backend  validates  launch    ↓  If  launch  succeeds:      load  main  simulator  app    ↓  If  launch  fails:
---- page 10 ----
load  fallback  error  page      allow  support  ticket  creation   
The  fallback  page  should:  
Fallback  error  page  ├──  display  a  clear  user-friendly  error  ├──  avoid  exposing  sensitive  technical  details  to  the  user  ├──  collect  minimal  diagnostic  metadata  ├──  allow  user  to  add  a  short  description  ├──  allow  “Contact  support”  immediately  ├──  create  a  support  ticket  even  when  the  main  app  fails  └──  route  the  issue  to  the  configured  support  email   
This  prevents  an  LTI  “catch-22”  where  the  student  cannot  access  the  application  and  also  
cannot
 
access
 
support.

This  is  especially  important  in  a  MERN-style  architecture  where  React  state,  backend

and
 
LMS

must
 
be

8.1  Global  Diagnostic  State  
The  React  frontend  should  maintain  a  global  diagnostic  snapshot.  
GlobalDiagnosticState  ├──  user  role  ├──  LMS  source:  Canvas  /  Moodle  /  Secure  Link  ├──  college  /  programme  /  course  identifiers  ├──  simulation  assignment  ID  ├──  attempt  ID  ├──  current  attempt  status  ├──  current  route/screen  ├──  microphone  permission  status  ├──  dictation  test  status  ├──  browser/device  information  ├──  network/API  status  ├──  last  successful  API  response
---- page 11 ----
├──  last  failed  API  response  ├──  last  AI  response  status  ├──  last  grade-sync  status,  if  relevant  ├──  timestamp  └──  client-side  error  summary   
When  the  user  clicks  Help ,  the  frontend  should  send  a  snapshot  of  this  diagnostic  state  to  
the

User  clicks  Help    ↓  Frontend  captures  diagnostic  snapshot    ↓  Snapshot  is  sent  to  backend    ↓  Backend  stores  searchable  metadata    ↓  Raw  diagnostic  log  is  stored  separately    ↓  Support  ticket  is  created  or  updated   
The  diagnostic  snapshot  should  support  both:  
1.  automated  troubleshooting;  2.  human  escalation  by  email.  
 
9.  Competency  Library  
The  system  should  include  a  layered  competency  library.  
Core  Competency  Library    ↓  College  Competency  Library    ↓  Programme  Competency  Library    ↓  Course  Competency  Library    ↓  Simulation-Specific  Criteria    
9.1  Competency  Levels
---- page 12 ----
Level  Created  /  Managed  By  Purpose  
Core  library  Platform  admin  /  academic  lead  
Shared  system-wide  competency  domains  
College  library  College  manager  Institution-specific  standards  
Programme  library  Programme  manager  Programme-specific  competencies  
Course  library  Teacher  Course-specific  teaching  emphasis  
Simulation-specific  criteria  
Teacher  /  programme  manager  
Criteria  relevant  only  to  one  simulation  
Teachers  should  be  able  to  add  competencies:  
1.  to  a  specific  simulation;  2.  to  a  specific  course;  3.  as  a  suggested  addition  to  the  programme  or  central  library.  
 
9.2  Competency  Governance  
Teacher-created  competencies  should  not  automatically  become  part  of  the  central  library.  
Recommended  workflow:  
Teacher  creates  local  competency    ↓  Uses  it  in  course  or  simulation    ↓  Optionally  submits  it  to  programme  library    ↓  Programme  manager  approves  /  edits  /  rejects    ↓  Optional  promotion  to  college  or  core  library   
Competency  statuses:  
Draft  Local  course  competency  Simulation-specific  criterion  Submitted  for  review  Approved  for  programme  Approved  for  college  Core  /  central  Deprecated
---- page 13 ----
9.3  Mapping  Local  Competencies  
Local  criteria  should  preferably  be  mapped  to  central  competency  domains.  
Example:  
Teacher-created  criterion:  “The  student  notices  that  the  client  avoids  contacting  friends.”   Mapped  to:  -  functional_analysis  -  avoidance_patterns  -  ACT_case_conceptualisation   
Important  rule:  
Mapped  competencies  →  appear  in  longitudinal,  course,  programme,  and  college  graphs   Unmapped  local  criteria  →  appear  only  in  the  specific  course  or  simulation  report   
This  allows  teacher  flexibility  while  preserving  meaningful  analytics.  
 
10.  Educator  Case  Authoring  Engine  
The  platform  should  include  a  semi-automated  case  authoring  system.  
Teachers  and  programme  managers  should  not  need  to  write  complex  AI  prompts  manually.  
Educator  Case  Authoring  Engine  ├──  Guided  simulation  builder  ├──  Prompt-driven  persona  generation  ├──  Diversity/context  engine  ├──  Clinical  ground-truth  boundaries  ├──  Hidden  information  rules  ├──  Resistance  mapping  ├──  Dynamic  emotional-state  settings  ├──  Challenge-level  settings  ├──  Multilingual  settings  ├──  Rubric  generator
---- page 14 ----
├──  Simulation  preview  ├──  Authoring  quality  checker  └──  Version  control   
No  batch  case  importer  is  included  in  this  HLD.

The  builder  should  guide  the  educator  through  structured  fields.

Mode  Practice  /  formative  /  graded  
Number  of  turns  50–75

10.2  Prompt-Driven  Persona  Generation  
The  system  should  generate  realistic  patient/client  personas  from  educator  inputs.  
The  persona  generator  should  create:
---- page 15 ----
●  demographic  profile;  ●  contextual  background;  ●  communication  style;  ●  emotional  presentation;  ●  cultural/contextual  considerations;  ●  values  conflict;  ●  avoidance  patterns;  ●  help-seeking  attitude;  ●  resistance  style;  ●  likely  disclosure  pattern.  
 
10.3  Inclusivity  Principle  
The  diversity/context  engine  should  support  varied  and  culturally  sensitive  personas,  while  
avoiding
 
stereotype-based
 
design.
 
Good  design:  
Cultural  or  demographic  context  →  influences  language,  life  context,  barriers,  values,    and  help-seeking  behaviour  only  when  educationally  relevant.   
Bad  design:  
Demographic  identity  →  automatically  determines  pathology,  personality,  or  risk.   
The  educator  should  be  able  to  review  and  edit  all  generated  persona  details  before  
publication.
 
 
11.  Clinical  Ground-Truth  Boundaries  
Every  simulation  should  include  a  structured  ground-truth  file.  
The  AI  patient  may  improvise  natural  language,  emotion,  hesitation,  and  style,  but  must  not  
invent
 
new
 
case
 
facts
 
outside
 
approved
 
boundaries.
 
Case  Ground  Truth  ├──  Known  facts  ├──  Presenting  problem  ├──  Symptoms  /  difficulties  ├──  Timeline
---- page 16 ----
├──  Functional  impairment  ├──  Maintaining  processes  ├──  Values  conflicts  ├──  What  the  patient  knows  ├──  What  the  patient  does  not  know  ├──  What  may  be  disclosed  ├──  What  must  not  be  invented  ├──  Risk  /  safeguarding  boundaries  └──  Escalation  rules   
Core  rule:  
The  AI  patient  can  vary  expression,  but  not  clinical  reality.   
This  is  essential  for  fair  assessment.  
 
12.  Hidden  Triggers  and  Resistance  Mapping  
The  simulation  should  include  programmable  behavioural  rules.  
The  patient  should  not  reveal  all  information  automatically.  Some  information  should  emerge  
only
 
when
 
the
 
student
 
demonstrates
 
appropriate
 
skill.
 
Example  disclosure  logic:  
If  student  validates  experience  →  patient  becomes  slightly  more  open   If  student  asks  a  specific  behavioural  question  →  patient  gives  more  concrete  detail   If  student  explores  values  respectfully  →  patient  reveals  value  conflict   If  student  gives  premature  advice  →  patient  becomes  more  guarded   If  student  pressures  the  patient  →  patient  changes  subject  or  becomes  defensive   If  student  misses  an  emotional  cue  →  patient  becomes  quieter  or  vague
---- page 17 ----
This  allows  the  simulation  to  assess  clinical  skill  rather  than  only  student  persistence.  
 
13.  Dynamic  Emotional-State  Tracking  
The  simulator  should  maintain  an  internal  patient  state  throughout  the  conversation.  
Patient  State  ├──  trust  ├──  openness  ├──  emotional  activation  ├──  avoidance  level  ├──  defensiveness  ├──  alliance  quality  ├──  disclosure  readiness  └──  risk/safety  relevance   
The  patient  response  should  depend  on  this  state.  
Example:  
Student  is  empathic  and  specific  →  trust  increases  →  patient  gives  richer  answer   Student  is  rushed  or  judgmental  →  defensiveness  increases  →  patient  becomes  brief,  vague,  or  evasive    
14.  Challenge  Level  
Teachers  should  be  able  to  define  how  challenging  the  patient  is.  
The  visible  control  can  be  simple:  
Challenge  level:  1–5   
Underneath,  the  challenge  level  should  affect  multiple  dimensions.  
Dimension  Low  Challenge  High  Challenge  
Openness  Shares  easily  Guarded  and  vague
---- page 18 ----
Emotional  intensity  Calm  Distressed  or  reactive  
Complexity  One  clear  issue  Multiple  interacting  issues  
Insight  Understands  patterns  
Confused  or  externalising  
Alliance  Cooperative  Tests  the  therapist  
Risk  ambiguity  None  Indirect  hints  
Disclosure  Reveals  quickly  Reveals  only  after  trust  
Structure  Follows  questions  Tangential  or  scattered  
Example:  
Challenge  Level  4  ├──  more  vague  answers  ├──  slower  disclosure  ├──  stronger  avoidance  ├──  more  emotional  shifts  ├──  more  indirect  risk  signals  └──  greater  need  for  precise  therapeutic  skill    
15.  Non-Verbal  Cue  Simulation  
The  AI  patient  should  be  able  to  include  controlled  non-verbal  cues  in  text.  
Examples:  
[Looks  away]  [Long  pause]  [Sighs]  [Smiles  nervously]  [Answers  quickly,  then  changes  the  subject]   
These  cues  should:  
●  appear  only  when  relevant;  ●  reflect  the  patient’s  emotional  state;  ●  be  visible  in  the  transcript;  ●  be  available  for  evaluation;  ●  not  become  excessive  or  distracting.  
Example  evaluation  use:
---- page 19 ----
The  patient  looked  away  and  became  vague  after  the  student  moved  too  quickly  into  advice.  The  student  did  not  notice  or  respond  to  this  cue.    
16.  Supplementary  Clinical  Information  Module  
Instead  of  focusing  on  medical  vitals  or  lab  tests,  the  platform  should  support  optional  
supplementary
 
information
 
relevant
 
to
 
psychotherapy
 
training.
 
Examples:  
●  referral  letter;  ●  previous  treatment  summary;  ●  symptom  questionnaire;  ●  risk  screening  summary;  ●  values  questionnaire;  ●  avoidance  inventory;  ●  school/work  functioning  note;  ●  attendance  or  engagement  history.  
Teachers  can  decide  whether  students  may  request  these  during  the  simulation.  
 
17.  Student  Simulation  Interface  
The  student  interface  should  support:  
Student  Simulation  Interface  ├──  Text  chat  ├──  Voice  dictation  ├──  Optional  AI  patient  audio  ├──  Timer  ├──  Pause  &  reflect  mode  ├──  Clinical  notes  side  panel  ├──  Formulation  side  panel  ├──  Simulation  instructions  ├──  Technical  support  button  ├──  Finish  simulation  button  └──  Post-simulation  feedback    
17.1  Text  and  Voice  Input  
The  core  interface  should  be  text-based,  with  optional  dictation.
---- page 20 ----
Recommended  flow:  
Student  speaks    ↓  Audio  is  transcribed    ↓  Text  appears  in  input  box    ↓  Student  can  edit    ↓  Student  sends  response   
This  gives  realism  while  preserving  student  control  and  accurate  assessment.  
 
17.2  Optional  AI  Patient  Audio  
In  a  later  phase,  the  patient’s  response  may  be  played  aloud  using  text-to-speech.  
Levels:  
Level  1:  Student  dictates;  patient  replies  in  text.   Level  2:  Student  dictates;  patient  replies  in  text  +  audio.   Level  3:  Full  real-time  voice  conversation.   
MVP  should  support  dictation  first.  Patient  audio  can  be  phase  2.  
 
17.3  Time-Bound  Sessions  
Teachers  should  be  able  to  configure  time  limits.  
Options:  
●  no  limit;  ●  15  minutes;  ●  30  minutes;  ●  50  minutes;  ●  custom.
---- page 21 ----
Assessment  mode  may  use  a  hard  stop.  Practice  mode  may  allow  pause.  
 
17.4  Pause  &  Reflect  Mode  
The  system  should  include  a  configurable  pause  mode.  
During  pause,  students  can:  
●  review  the  conversation;  ●  write  notes;  ●  draft  the  next  question;  ●  consult  an  AI  tutor  if  allowed;  ●  reflect  on  formulation;  ●  plan  next  steps.  
Mode  Pause  Allowed?

18.  Simulation  Runtime  Model  
The  simulation  should  not  work  as  a  simple  chatbot.  
Instead,  it  should  use  a  structured  runtime  process.  
Student  message    ↓  Interaction  analyser    ↓  Update  patient  state    ↓  Check  ground-truth  and  disclosure  rules    ↓  Check  resistance  /  trigger  rules    ↓  Generate  patient  response    ↓  Store  transcript  +  state  changes   
The  interaction  analyser  should  evaluate:  
●  empathy;  ●  question  type;  ●  specificity;  ●  validation;  ●  ACT  consistency;  ●  premature  advice;  ●  pressure;  ●  missed  cues;  ●  risk/ethics  relevance;  ●  therapeutic  stance.  
The  result  affects  the  patient’s  next  response.  
 
19.  Rubric  Builder  
Each  simulation  should  have  a  semi-automatically  generated  rubric.
---- page 23 ----
The  rubric  should  be  built  from:  
●  simulation  goals;  ●  clinical  model;  ●  student  level;  ●  challenge  level;  ●  central  competency  library;  ●  programme  competencies;  ●  teacher-added  criteria;  ●  simulation-specific  criteria.  
Rubric  flow:  
Teacher  defines  simulation    ↓  System  suggests  rubric    ↓  Teacher  edits  criteria,  weights,  and  wording    ↓  System  generates  scoring  anchors    ↓  Teacher  approves    ↓  Rubric  version  is  locked  for  assigned  simulation   
Example  rubric:  
Criterion  Source  Weight  
Builds  therapeutic  alliance  Core  library  15  
Clarifies  the  presenting  problem  Core  library  15  
Identifies  avoidance  patterns  Programme  ACT  library  15  
Uses  ACT-consistent  stance  Programme  ACT  library  20  
Explores  values  Core  /  ACT  library  10  
Notices  risk  /  ethical  boundaries  Core  library  10  
Structures  the  session  effectively  
Core  library  10  
Responds  to  guardedness  Simulation-specific  5   
20.  Evaluation  and  Debriefing  Engine
---- page 24 ----
At  the  end  of  the  simulation,  the  system  should  generate  structured  evaluation.  
Evaluation  Engine  ├──  Rubric  scoring  ├──  Criterion-level  feedback  ├──  Transcript  evidence  ├──  Highlighted  strengths  ├──  Highlighted  missed  opportunities  ├──  Student-facing  summary  ├──  Teacher-facing  report  ├──  Remediation  suggestions  ├──  Student  debrief  chat  └──  Optional  Canvas/Moodle  grade  return   
The  system  should  first  generate  structured  data,  then  language.  
Example  structure:

anxiety

20.1  Transcript  Highlighting  
The  report  should  highlight  specific  moments  in  the  conversation.  
Examples:  
Strong  moment:  Turn  8  —  Student  asked  an  open  and  non-leading  question.   Missed  opportunity:  Turn  17  —  Patient  hinted  at  avoidance,  but  the  student  moved  directly  to  advice.   Risk/ethics  flag:  Turn  24  —  Patient  made  an  indirect  safety-relevant  statement  that  was  not  explored.   
This  makes  feedback  transparent  and  teachable.

20.3  Post-Simulation  Debrief  Chat  
After  receiving  feedback,  the  student  should  be  able  to  chat  with  a  debriefing  tutor.  
This  is  not  the  patient.  It  is  an  educational  supervisor.  
Students  can  ask:  
Why  did  I  lose  points  on  functional  analysis?  What  would  have  been  a  better  question?  Where  did  the  client  hint  at  avoidance?  How  could  I  have  explored  values  better?  Show  me  an  example  of  a  stronger  response.  Did  I  miss  any  risk  indicators?
---- page 26 ----
Guardrails:  
-  uses  only  transcript,  rubric,  and  evaluation;  -  does  not  invent  new  clinical  facts;  -  does  not  change  the  official  grade;  -  explains  feedback  pedagogically;  -  can  cite  relevant  transcript  moments;  -  can  be  visible  to  the  teacher  in  graded  mode.    
20.4  Remediation  Suggestions  
The  system  should  suggest  targeted  learning  based  on  weak  competencies.  
Examples:  
If  weak  in  functional  analysis:  →  suggest  micro-learning  on  triggers,  behaviours,  and  consequences.   If  weak  in  values  exploration:  →  suggest  values  clarification  examples.   If  weak  in  risk  awareness:  →  suggest  review  of  appropriate  risk/safety  inquiry  within  training  scope.   If  weak  in  therapeutic  stance:  →  suggest  examples  of  validation  and  open  questions.   
Remediation  can  include:  
●  short  reading;  ●  flashcards;  ●  example  questions;  ●  practice  simulation;  ●  teacher-assigned  review  task.

38.  Core  Design  Principles   
38.1  Hierarchy  First  
The  system  must  be  designed  from  the  start  around:  
College  →  Programme  →  Course  →  Simulation  →  Student  Attempt   
This  affects  permissions,  dashboards,  grades,  analytics,  credits,  reporting,  and  support  
routing.
 
 
38.2  Competency  Mapping  Is  Essential  
Teacher  flexibility  is  important,  but  analytics  require  shared  competency  mapping.  
Flexible  local  criteria  +  central  competency  mapping  =  meaningful  longitudinal  analytics    
38.3  The  AI  Patient  Must  Have  Boundaries  
The  patient  can  improvise  wording  and  emotion,  but  not  invent  clinical  facts  outside  the  
ground-truth
 
file.
 
 
38.4  Simulation  Is  More  Than  Chat  
The  system  should  track  patient  state,  trust,  disclosure  readiness,  resistance,  emotional  
activation,
 
and
 
non-verbal
 
cues.
 
 
38.5  Feedback  Must  Be  Evidence-Based  
Every  evaluation  should  connect  to  transcript  evidence.
---- page 65 ----
38.6  Student  Debriefing  Is  Part  of  Learning  
The  post-simulation  evaluator  chat  should  help  students  understand  their  performance  
without
 
changing
 
the
 
official
 
grade.
 
 
38.7  Multilingual  Support  Must  Be  Structural  
Translation  should  not  be  an  afterthought.  Competency  IDs,  rubric  IDs,  scores,  and  analytics  
must
 
be
 
language-independent.
 
 
38.8  Formal  Assessment  and  Personal  Practice  Must  
Remain
 
Separate
 
Future  student-purchased  credits  should  support  extra  learning,  but  not  distort  official  course  
grades.
 
 
38.9  Failure  Must  Be  Supported  Before  the  App  Boots  
LMS  launch  failures  may  occur  before  the  main  frontend  loads.  The  platform  must  therefore  
provide
 
a
 
fallback
 
error
 
page
 
capable
 
of
 
creating
 
support
 
tickets
 
independently
 
of
 
the
 
main
 
React
 
app.
 
 
38.10  Technical  Support  Must  Be  Separate  From  
Clinical
 
Simulation
 
Technical  support  must  not  share  the  AI  patient’s  prompt,  transcript  context,  case  facts,  or  
clinical
 
roleplay
 
state.
 
 
38.11  Deterministic  Support  First  
For  MVP,  support  should  rely  on  predictable  troubleshooting  flows  and  escalation  rules,  not  
open-ended
 
generative
 
support.
---- page 66 ----
38.12  Always  Provide  a  Human  Escape  Hatch  
Users  should  always  be  able  to  skip  directly  to  email  support.  
 
38.13  Technical  Failure  Must  Not  Become  Academic  
Penalty
 
If  a  simulation  is  disrupted  by  confirmed  technical  failure,  the  attempt  should  be  marked  
accordingly
 
and
 
the
 
student
 
should
 
be
 
protected
 
from
 
unfair
 
grading
 
consequences.
 
 
38.14  Support  Privacy  Is  Essential  
Technical  support  should  not  receive  full  transcripts,  evaluations,  clinical  notes,  hidden  case  
material,
 
or
 
student
 
reflections
 
by
 
default.
 
 
38.15  Grade  Sync  Must  Be  Idempotent  
Manual  or  automated  grade-sync  retries  must  not  create  duplicate,  conflicting,  or  repeated  
LMS
 
grade
 
submissions.
 
 
38.16  Logs  Must  Be  Searchable  but  Not  Bloated  
Support  tickets  should  contain  searchable  metadata.  Large  raw  diagnostic  logs  should  be  
stored
 
separately
 
with
 
redaction
 
and
 
retention
 
controls.
 
 
38.17  Cost  Governance  Is  a  Core  Requirement  
AI  usage  must  be  visible,  bounded,  and  auditable  at  student,  course,  programme,  and  
college
 
levels.
