## [win] Independent review gate (Oren) caught a production-only crash bug in guard-runner -- 2026-07-11
Source: projects/ai-patient-simulator/docs/review-sprint-5-oren.md (M1 finding and FIX STATUS section)

Oren's Sprint 5 review returned APPROVE-WITH-CONDITIONS (0 blocker, 2 major, 4 minor, 4 info).
The most significant finding was M1.

In packages/engine/src/pipeline/guard-runner.ts, the operator precedence in the verdict check
was wrong: the expression was (A && B && C && D) || E instead of A && B && C && (D || E). On
the FAIL branch, the code could enter without the null guards in place. If a real LLM guard
returns {"verdict":"FAIL"} without a violations array, the subsequent violations.join() call
throws a TypeError and crashes turn processing -- exactly on the path where correctness matters
most. Under StubProvider the bug is completely invisible, because the stub always returns a
well-formed object. It would only surface at APS-004 production go-live, the first time a real
model is wired into the guard path.

Eco fixed the bug same session, going further than Oren's parenthesize-only recommendation:
violations and suggestion fields are now defaulted when absent on a well-formed verdict object,
closing the crash path entirely even if a real model returns a partial response. Two regression
tests were added to the engine test harness.

The important fact: the engineering team signed off their work; unit and integration suites
passed; the E2E ran clean. None of that could find an operator-precedence bug whose failure mode
depends on a real LLM returning a specific but valid response shape. A human reviewer reading
the code found it.

## [win] Sami C2 compounding-invented-facts test proved the ground-truth boundary structurally -- 2026-07-11
Source: projects/ai-patient-simulator/docs/qa-arc-coherence-signoff-adi-2026-07-11.md (Test 2 -- C2-INVENTED-FACTS)

Sami's C2 required proving that invented facts from a session summary cannot propagate into
authoritative ground truth or guard model input. Adi built this as a failing-able integration
test against live Postgres.

Setup: seed a controlled false fact into the session-2 ArcSessionSummary ("patient mentioned a
sister" -- absent from authored ground truth). Load session 3. Critical assertion (C2-T3): the
false fact must NOT appear in the guard prompt or the ground-truth input. A pass proves the
boundary; a fail means the architecture broke. C2-T3 passed.

The architectural property that holds this: buildGuardPrompt() receives only the authored
SimulationTemplate groundTruth. It has no arcContext parameter. The arc block is injected into
the patient context builder as a system message labeled "PRIOR SESSION CONTEXT -- context only,
not ground truth." The false fact is bounded to that block and cannot be promoted to authoritative
ground-truth status. If a future refactor accidentally routes summary content into the guard input,
C2-T3 will catch it before the sprint closes.

## [pattern] Independent review gate exists to catch the class of bug that stubs cannot find -- 2026-07-11
Source: projects/ai-patient-simulator/docs/review-sprint-5-oren.md (M1 finding)

Oren's Sprint 5 review found a crash bug that was entirely invisible under StubProvider. The
bug would only surface at APS-004 production go-live, the first time a real LLM is wired into
the guard path. Tests passed; E2E passed; code review by the build team passed. A reviewer
reading the logic found an operator-precedence error that the test suite structurally could not
reach.

The gate is currently Oren reviewing every sprint as an independent reviewer who does not build.
The structural lesson: the independence matters. If sprint pressure ever pushes toward "skip the
review, the tests passed," this finding is the documented reason to resist that.

## [pattern] Clinical conditions operationalized as failing-able integration tests -- 2026-07-11
Source: projects/ai-patient-simulator/docs/qa-arc-coherence-signoff-adi-2026-07-11.md (Test 2)

Sami's C2 condition was stated as a clinical requirement about information boundaries. Adi
built it as a concrete integration test that fails if the architectural boundary is violated.
A clinical condition stated in prose is only as durable as the team's memory of it. A failing
test is durable across every future sprint and every future refactor.

This is now the precedent on the APS project: Sami's conditions do not remain advisory
documentation. They become assertions in the integration suite. C2-T3 will catch a boundary
violation the next time it occurs, regardless of who made the change and whether they remembered
the condition.
