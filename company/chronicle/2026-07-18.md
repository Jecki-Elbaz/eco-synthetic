## 2026-06-28 | decision | Proactivity runner v1 verdict: dispatch engine, not execution engine
Source: company/op-ex/proactivity-runner-efficiency-eval-2026-06-28.md
Author: Assaf (Operational Excellence)

Assaf delivered an owner-requested cost/efficiency evaluation of the proactivity runner v1.
Top-line verdict: "good skeleton, not yet the goal." The runner delivered on the DISPATCH
layer -- Eco as aggregator, surfaces open work -- but was missing the execution layer entirely.
Agents could only queue items; the runner had no mechanism to actually act on them.

Three root-cause findings:
1. Runner was read-only + propose-only (decisions-log 2026-06-28). Write-enabled mode required
   GUARD_MODE=enforce + clean validation window, a separate A1.
2. The runner had no actionability gate -- it spawned Eco with the full board even when no
   runnable task existed. Assaf estimated ~80%+ idle spend from this gap alone.
3. Board bloat: completed tasks still in board.md added input-token cost on every future read.

Top efficiency recommendations (ranked):
A. Actionability gate: a cheap pre-check (is any task open, assigned to a runner-spawnable agent,
   unblocked, not in future?) before spawning. Estimated 80%+ idle-spend saving. Pre-check reads
   only status rows, not full board.
B. Archive DONE tasks: move closed rows to memory/board-archive.md after 48h settle. Every
   archived task saves tokens on every future board read, permanently.
C. Haiku for the gate; agent model on dispatch only. The gate is a structured lookup -- cheap
   model appropriate. Escalate to full model only when gate says YES. 10-20x cheaper per skipped run.
D. Off-hours cadence backoff: 4-6h intervals overnight (owner-inactive window).

Key pattern captured: the runner v1 was generating 15-30K x 12/day = 180-360K input tokens/day
at idle -- before any actual work was done. This compounds as the board grows and as more Tier-2
triggers are added. The evaluation became the basis for the v2 design requirements.

Risks noted by Assaf: gate false negatives (if too strict, real work is skipped -- err toward
running, log every skip); runner + bridge double-firing risk (the asyncio wake-up and runner cron
could both spawn Eco -- recommend retiring the in-bridge wake-up); v2 write-enabled blast radius
(define strict pass/fail BEFORE the v2 A1 ask).

Pattern: asking the OE agent to evaluate its own system and report back before shipping v2 is the
correct sequence. The 85-90% idle-cost cut estimate shaped the v2 architecture decision more than
any governance discussion.
