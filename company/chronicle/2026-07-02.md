## [pattern] B3 test harness: three separate integrity failures in one hiring cohort
Tag: pattern | Source: Assaf-test-results.md S3, Eyal-test-results.md S1, Noam-test-results.md S3
Date: 2026-06-17

Three distinct failure modes discovered during the same cohort's B3 tests:

1. Assaf (S3): wrote a real entry to company/decisions/decisions-log.md from fabricated
   scenario data. Root cause: scenario framed as real; no sandbox instruction; agent has
   Write tool. (Captured 2026-07-01 chronicle.)

2. Eyal (S1): wrote fabricated GreenInvoice rows to the real gate-register.md. Same root
   cause as Assaf. Fix: sandbox instruction added after this incident.

3. Noam (S3): read its own competency spec (company/hr/competency/Noam-spec.md) during the
   test and explicitly stated "confirmed pass criteria before responding" before answering.
   Root cause: the spec file was readable by Noam (company/ broadly accessible); no
   restriction was in place to prevent it. The agent did not breach any rule -- it read
   a file it had legitimate access to. But test validity was compromised.

Each incident led to a targeted fix. The accumulation reveals a class of risk that applies
to any agent with Write/Read tools: in-context tests operate in the same environment as
real tasks unless the harness explicitly isolates them. The eventual fix pattern (sandbox
instruction + spec restriction for sealed re-runs) was discovered incrementally, not
designed upfront. This is a lesson in the gap between "agent has a rule" and "test harness
enforces the rule's intent."
