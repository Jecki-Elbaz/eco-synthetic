### [mistake] Spec vs. role-file conflict -- Shir Scenario 1 rollback authority (2026-06-16)

Source: Shir-anat-review.md, Shir-conditions-resolution.md.

The Shir-spec.md Scenario 1 pass criteria erroneously required pre-authorization before
a code-only active-incident rollback. The role file (Shir.md) correctly grants A2
authority for exactly this case as "decide-and-log," not "pre-authorize."

The spec and the role file directly contradicted each other on one of the most
operationally important judgment calls for a DevOps agent. Eco adjudicated, owner approved
the correction at A1, spec was updated, Shir re-scored against the corrected criteria.

Lesson: competency specs must be written AFTER the role file is final and checked against
the role file's authority matrix before use. The spec evaluates whether the agent matches
the role, not whether the role matches some independent ideal. When specs are written in
parallel with or before role files are locked, conflicts like this are predictable.

### [pattern] S5 -- the missing destructive-command list is a cross-role gap (2026-06-16)

Source: Shir-anat-review.md.

Rambo's scan of Shir flagged that the CLAUDE.md red line 3 destructive-command list
(rm -rf, DROP TABLE, git push --force to main, git reset --hard on shared branches, any
data-deletion operation) was absent from Shir's "What you must NEVER do" section. Gal's
role file had this enumeration; Shir's did not.

Anat upgraded this to condition S5 (not just a non-blocking note) because Shir holds the
most operationally powerful Bash in the R&D group -- production infra deploy, rollback,
integrations/ write. The judgment was sound (demonstrated in Scenario 2), but role files
must bind the constraint in writing, especially for the highest-blast-radius tools.

Pattern: for any agent holding Bash with state-changing infra scope, the destructive-
command enumeration is mandatory in the never-do section, not optional. Auditing future
Bash-holding agent role files against this list should be on Rambo's R&R checklist.

Source: Designer-rambo-scan.md.

Designer's scan (CLEAR-WITH-CONDITIONS) introduced F2 (Write scope not bounded): the
role file named write targets (projects/delivery-saas/docs/) but contained no explicit

in the role file is the primary security control equivalent to destructive-command
enumeration for Bash-holding agents.
