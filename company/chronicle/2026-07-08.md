## [decision] Rambo flags Bash as excess for VP R&D (Ido) -- A1 removal required
Tag: decision | Source: company/hr/competency/Ido-rambo-scan.md, Ido-test-results.md B7
Date: 2026-06-17

Rambo's B5 scan of Ido found Bash as a high-severity excess permission. Ido's role --
R&D planning, release-gate decisions, sprint prioritization, architecture escalation -- has
zero responsibilities that require shell execution. Code execution is explicitly delegated
to Gal (dev validation) and Shir (infra/deploy). Ido reads output artifacts, does not run
them. Finding is consistent with T-0020 follow-up assessment (2026-06-16).

Consequence: Ido blocked from certification until Bash removed (A1 by jecki, coordinated
by Eco). This cascaded: Gal and Shir needed Ido live for B6 sign-off, so the Ido Bash
condition was the critical-path blocker for the entire R&D group go-live.

The underlying principle: tool grants must track explicit role responsibilities, not seniority
or past practice. A VP needing to "be informed" about code does not need to run it.

## [pattern] Hiring pipeline cascade: Ido activation is the R&D group keystone
Tag: pattern | Source: company/hr/competency/Ido-test-results.md B7, Gal-test-results.md B7
Date: 2026-06-17

Ido's B7 go-recommendation explicitly noted the cascade: "Gal and Shir B6 sign-off
requires Ido live. Both are blocked until owner resolves condition 1 and activates Ido."
Condition 1 was the Bash removal (A1 by jecki).

A1 call), but the pattern is worth noting for future hiring cohorts.
