## 2026-07-19 | win | Runner silent-failure root-caused and fixed -- cmd.exe 8191-char limit
Source: integrations/runner/runner.py (ECO-CMDLINE-FIX comment block) + commit b920e8f
Author: Eco (root-cause) + commit (Eco/owner)

For ~36 hours (2026-07-17/18) runner jobs were failing silently. rc=1, empty stdout --
but the runner was logging them as clean "done" events. No Telegram alert, no error_final,
no visible sign anything was wrong.

ROOT CAUSE (two compounding bugs):

1. WRONG BINARY: the runner was calling `claude.cmd`, a batch shim that routes through
   cmd.exe. cmd.exe enforces an 8191-character command-line limit. The runner passes the
   full agent role file as an --append-system-prompt ARGUMENT. Role files are ~8000 chars.
   When the per-job Gmail tools list was added to the command line, total argv crossed the
   limit. cmd.exe returned "The command line is too long." on stderr, rc=1, empty stdout.

2. SILENT FAILURE LOGGING: the run_invoke() function treated rc=1 as a clean "done" when
   stderr matched no known retry pattern. So the failure produced no alert, no retry, no
   escalation -- just a silent null result logged as success.

FIXES (two-part):
- find_claude() now prefers claude.exe (direct CreateProcess, 32767-char limit) over the
  .cmd shim, removing the failure mode entirely.
- run_invoke() now treats any nonzero exit as a failure even when stderr matches no retry
  pattern (returns err_tag "rc=N: <stderr>"), so future unknown failures surface immediately.

Pattern: a runtime failure that produces no error signal and logs as success is the worst
class of operational bug -- it can run undetected indefinitely. The fix closes both the
immediate failure mode AND the silent-logging gap that made it invisible. The 36-hour window
is the cost of having no alerting on nonzero exit codes.
