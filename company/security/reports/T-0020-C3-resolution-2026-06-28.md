# Security Memo: T-0020 C3 Resolution -- Shell-Tool Stripping via Scheduled Runner

Author: Claude Code session, owner (jecki) A1. Date: 2026-06-28.
Resolves: T-0020 condition C3 (and the binding R2 blast-radius constraint) for the

---

## 1. What C3 required

T-0020 R2 (binding risk): an auto-spawned Bash-capable agent (Gal, Shir, Adi) could run
shell commands reachable from an untrusted trigger. C3: confirm settings.json deny rules
cascade to spawned subprocesses, OR provide a compensating control -- "shell-tool stripping."
The report states: "once shell-tool stripping is in place, C3 is resolved by architecture;
blast radius from the bridge is bounded regardless of the spawned agent's role-file toolset."

## 2. The resolution: a dedicated stripped runner spawn-path

Proactivity agents are launched by a SCHEDULED RUNNER (shared/scripts/agent-runner.py), NOT
by the Telegram bridge and NOT via the in-session Agent tool. Properties:

- TOOL STRIPPING (enforces R2/C3): the runner launches each agent as a separate headless
  `claude --print` process with an EXPLICIT allowed-tools whitelist. The whitelist is
  Read (readonly mode) or Read,Write,Edit (act mode) -- and NEVER Bash, WebFetch, or
  WebSearch. Because the runner sets --allowedTools per launch, the agent's role-file
  toolset is irrelevant; a Bash-holding agent (Eco, Gal, Shir) runs WITHOUT Bash here.
- DENY-CASCADE MOOT: since Bash is never granted on this path, the open C3 question
  ("do deny rules cascade to subprocesses") is moot for runner spawns.
- NO EXTERNAL INPUT (mitigates R1/R5): the runner is TIMER-triggered. The task envelope
  comes from the schedule + the board, not from a Telegram message. The prompt-injection
  vectors that gate the bridge path do not apply here.
- AUDIT (C8): every run is appended to memory/agent-runs.jsonl (start/done/skip/error,
  agent, mode, tools, rc, summary).
- KILL SWITCH (section 7): the runner checks memory/SAFE_MODE before every launch and
  aborts (fail-safe: aborts if it cannot read the flag).
- GUARD BACKSTOP: the PreToolUse guard (.claude/hooks/guard.py) still evaluates every
  write inside each agent session (Red paths, append-only, SAFE_MODE).

## 3. What is and is NOT changed

- The in-session / Telegram-bridge Agent-tool allow-list (agent-tool-spawn-allowlist.md
  PERMITTED list; guard.py ALLOWED_AGENTS) is UNCHANGED and stays conservative. Stripping
  is only enforceable on the runner path, so the bridge path keeps its existing posture --
  no R2 hole is opened there.
- NEW: a separate "Runner-spawn (stripped)" list authorizes the runner to launch the
  reports (Gal, Shir, Adi, etc.) with Bash/Web stripped.

## 4. Residual / still gated

- AUTONOMOUS BASH (run tests, deploy, infra) remains BLOCKED. It is not granted on the
  runner path. Enabling it requires either confirming deny-cascade or building a sandbox
  -- a separate later phase, separate A1.
- GUARD_MODE stays SHADOW during the validation window (SEC-0001). Runner v1 runs
  READ-ONLY. Flipping to act (Write,Edit) + GUARD_MODE=enforce is a separate A1 after a
  clean validation window.

## 5. Verdict

C3 RESOLVED for the autonomous (runner) spawn path via shell-tool stripping. Reports may be
runner-spawned stripped. Bridge/in-session posture unchanged. Autonomous Bash still gated.
